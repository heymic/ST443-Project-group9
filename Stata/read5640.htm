DATA PROCESSING NOTES


Data Archive Processing Standards

The data were processed to the UK Data Archive's A* standard. This is the Archive's highest standard, and means that an extremely rigorous and comprehensive series of checks was carried out to ensure the quality of the data and documentation.�Firstly, checks were made that the number of cases and variables matched the depositor's records. Secondly, checks were made that all variables had comprehensible variable labels and all nominal (categorical) variables had comprehensible value labels. Where possible, either with reference to the documentation and/or in communication with the depositor, labels were accordingly edited or created. Thirdly, logical checks were performed to ensure that nominal (categorical) variables had values within the range defined (either by value labels or in the depositor's documentation). Lastly, any data or documentation that breached confidentiality rules were altered or suppressed to preserve anonymity.

All notable and/or outstanding problems discovered are detailed under the 'Data and documentation problems' heading below.

Useful Notes

Changes to the GHS methodology:
Users should note that during 2005, the GHS was in the process of changing from financial year collection (April to the following March) with a cross-sectional design to calendar year collection (January to December) with a longitudinal design. The GHS 2005 file is therefore drawn from two sources. The new design was introduced from April 2005 with a larger sample and the aforementioned calendar year cycle in the second quarter; these data are therefore available for quarters 2-4. Data for the first quarter are drawn from the GHS 2004-2005 in order to generate a file that relates to the full calendar year.

As a consequence of the change in design, a full year of interviews was carried out between April to December 2005; these cases were then re-allocated between January and December of 2006 for re-interview. From 2006, the GHS design was therefore on a calendar year basis and persons were re-interviewed for the first time. That is, only the April to December cases in 2005 are part of the longitudinal design. Further information about these changes may be found in the documentation (the GHS 2005 Overview Report).

Variables removed:
The following string variables are not included in the UK Data Archive version of the GHS 2005 dataset in order to preserve confidentiality: nb7idesc, lmat, lmat2, lmat3, lmat4, lmat5, lmat6, lmatter, cigfdesc, nb7fdesc, nbrfdesc, nbridesc, sb7fdesc, sbrfdesc, sbridesc, othsour2, findd, fjbdesc, fjbtitle, mindd, mjbdesc, mjbtitle, svisedf, svisedm, svisedsc, model, model2, model3, model4, model5, model6, model7, model8, crysp, crysp1, crysp2, cmatter, secondjb, and thirdjb. The variables include some string variables from the Social Mobility trailer, and users should note that they may still be referenced in the documentation.

Parents' SOC2000 variables from Social Mobility trailer:
Two variables from the social mobility trailer, SOC2KF (SOC2000 for father) and SOC2KM (SOC2000 for mother), which were not included in the first deposit, were added to the individual file in June 2007. Users should download a new copy of the dataset if they wish to use these variables in their analysis. However, users should be aware that as the GHS was in the process of changing from a financial to calendar year data collection period in 2005 (see above), the January to March collection did not include the trailer module, meaning that there are no SOC2KF or SOC2KM data for these cases, leaving only 25,093 eligible cases overall. Furthermore, the Social Mobility trailer was only asked of 25-65 year-olds. Therefore, many cases of SOC2KF and SOC2KM have no response.

Weight05 issue - resolved November 2007:
A problem was identified with the first edition of the GHS 2005 individual file, whereby 8,334 cases had the weight variable Weight05 set to SYSMIS. The cases affected were those under 16, or proxy respondents, so analysis of variables that include those types of respondent may be affected. A corrected file was issued for the second edition, in November 2007. The new edition also contained some variable changes, deletions and additions, which are listed in the 'New Edition Information' section above.

January 2010 update:
New HSERIAL, PID and FSERIAL values have been created using lookup tables of sequential numbers provided by ONS. The newly created variables HSERIAL, PID and FSERIAL have replaced the old variables. This update was made because previously cases for the first quarter did not have the person identifier variable PID, and it was not possible to uniquely identify individuals by combining information on HSERIAL and PERSNO, as duplicates occurred as a result of the changes in survey design (please see 'Changes to the GHS methodology' above).

Data conversion information

From January 2003 onwards, almost all data conversions have been performed using software developed by the UKDA. This enables standardisation of the conversion methods and ensures optimal data quality. In addition to its own data processing/conversion code, this software uses the SPSS and Stat/Transfer command processors to perform certain format translations. Although data conversion is automated, all data files are also subject to visual inspection by a UKDA data processing officer.

With some format conversions data, and more especially internal metadata (i.e. variable labels, value labels, missing value definitions, data type information), will inevitably be lost or truncated owing to the differential limits of the proprietary formats.�A UKDA Data Dictionary file (in rich text format), corresponding to each data file, is usually provided for viewing and searching the internal metadata as it existed in the originating format. These files are called: [data file name]_UKDA_Data_Dictionary.rtf

SPSS (*.por)

SPSS portable (*.por files)
If SPSS portable was not the ingest format, this format will generally either have been created via the SPSS command processor (e.g. if the ingest format is SPSS .sav, SAS, Excel, or dBase), or if the ingest format was STATA, the SPSS version will be created via the Stat/Transfer command processor. If the ingest format was undelimited text, the data will have been read into SPSS using an SPSS command file.

Issues: There is very seldom any loss of data or internal metadata when importing data files into SPSS. Any problems will have been listed above in the Data and Documentation Problems section of this file.

STATA (*.dta)

STATA (*.dta files)
If STATA was not the ingest format, all STATA files will have been created from SPSS .sav format via the Stat/Transfer command processor. Importantly, Stat/Transfer's optimisation routine is run so that variables with SPSS write formats narrower than the data (e.g. numeric variables with 10 decimal places of data formatted to FX.2) are not rounded upon conversion to STATA because they are converted to 'doubles ' rather than floats. User missing values are copied across into STATA (as opposed to being collapsed into a single system missing code).

Issues: There are a number of data and metadata handling mismatches between SPSS and STATA. Where any data or internal metadata has been lost or truncated, this will have been automatically logged in this file: 5640_SPSS_to_STATA_conversion.rtf Note that the complete internal metadata has been supplied�in the UKDA Data Dictionary file(s): [data file name]_UKDA_Data_Dictionary.rtf
